# V5.13 — Speaker identity bug fixes + Hero image fit option

## What's in this milestone

Three bug fixes reported by user on 2026-06-30 against the speaker-intro
mockup editor at /admin/mockups/speaker-intro.

### Bug 1: Speaker role change doesn't apply
**Symptom**: When trying to change a speaker from "Panelist" to "Speaker"
via the Role select in the form sidebar, the change doesn't apply — the
select appears to revert to its previous value.

### Bug 2: Order number writes wrong value, on multiple speakers
**Symptom**: When pressing "5" in a speaker's Order field, the value "7"
appears instead, AND it appears simultaneously in multiple speakers'
Order fields.

### Bug 3: Hero image crops to box size, doesn't fill when scaled
**Symptom**: When selecting a new hero image (e.g. the brand asset at
https://uojldinyokysycfc.public.blob.vercel-storage.com/brand-assets/1782656127534-p7pu9m.png),
the image is cropped to the original container size. Even when scrolling
out (zoom < 1) or enlarging the container (imageScale up), the crop
remains and the image is not full.

## Root causes

### Bugs 1 & 2 — same root cause: unstable speaker identity

The form view identified speakers by their `order` field using
`find(s => s.order === sp.order)` in every onChange handler. This is
unsafe because:

1. **Order values can collide.** If the user types "5" when another
   speaker already has order=5 (typo or prior corruption), there are now
   two speakers with order=5. `find` returns the first match — which may
   be a different speaker than the one the user is editing. The edit
   lands on the wrong speaker.

2. **The closure's `sp.order` is stale.** When the user types a new
   order value, the speaker moves in the sorted list. But the closure
   inside onChange still references the old `sp.order` from when the
   SubCard rendered. `find(s => s.order === <old order>)` returns the
   wrong speaker after the move.

3. **SubCard key was based on sorted index.** `key={`spk-${idx}`}`
   where `idx` is the index in the sorted array. When sort order
   changes, the SubCards shuffle their keys, causing React to unmount
   and remount the input the user is typing in — losing focus and
   creating the visual impression that "5 became 7" or that multiple
   speakers' fields updated at once.

### Bug 3 — `object-cover` always crops + placement carries over

The hero image always used `object-cover`, which scales the image to
fill the container, cropping any overflow. Even when the user zoomed
out (transform: scale < 1) or enlarged the container (imageScale up),
`object-cover` would re-cover the new container size, so the crop
remained.

Additionally, when the user picked a new hero image, the previous
image's `imagePlacement` (focusX, focusY, zoom) carried over to the
new image. If the previous image had been zoomed in, the new image
would start zoomed in too — making it look "cropped to the original
size of the box".

## Fixes applied

### Bugs 1 & 2 — use original array index as stable identity

In `speaker-intro-form-view.tsx`:

- Changed `speakersSorted` from `[...data.speakers].sort(...)` to
  `data.speakers.map((sp, origIdx) => ({sp, origIdx})).sort(...)`.
  The original array index is stable across re-sorts because we only
  ever push to the end of `data.speakers`, never insert in the middle.

- Changed the SubCard `key` from `spk-${idx}` (sorted index) to
  `spk-${origIdx}` (original index). Now when sort order changes, the
  SubCard keys stay stable — React no longer remounts inputs.

- Changed all 9 onChange handlers (Order, Role, Full name, Title,
  Company, Bio, Session title, Session time, Photo URL, Photo size,
  Visible) from `find(s => s.order === sp.order)` to direct index
  access `d.speakers[origIdx]`. This never returns the wrong speaker.

- Changed the delete handler from
  `d.speakers = d.speakers.filter(s => s.order !== sp.order)` (which
  would delete ALL speakers with the same order value) to
  `d.speakers.splice(origIdx, 1)` (deletes only the targeted speaker).

### Bug 3 — add `fit` option + reset placement on image change

- Added `fit?: "cover" | "contain"` field to `heroOverlay` type in
  `types.ts`. Default is "cover" (no behavior change for existing
  data). When set to "contain", the image is rendered with
  `object-contain` instead of `object-cover` — the full image is
  visible inside the container, letterboxed (transparent bars) if the
  aspect ratios differ.

- Added an "Image fit" select dropdown in the Hero image & overlay
  form section with two options:
  - Cover — fill container, crop overflow (default, best for landscape
    photos)
  - Contain — fit entire image inside container, letterbox if needed
    (best for brand assets / logos)

- Passed `fit` through to `EditableImage` via the `objectFit` prop in
  `speaker-intro-canvas.tsx`. The `EditableImage` component already
  supports both `object-cover` and `object-contain` (the className is
  chosen based on the `objectFit` prop), so no changes were needed
  inside `EditableImage` itself.

- In `speaker-intro-editor.tsx`, when a new hero image is picked
  (different URL from the previous one), now also resets
  `imagePlacement` to `{focusX: 50, focusY: 50, zoom: 1}`. This
  ensures the new image starts at default pan/zoom, not inheriting
  the previous image's crop.
