# v3.0.0 Tag + GitHub Release Procedure

> Plan ref: AISalon-Team-Plan-V3.0 §5.2 Backup Procedure, §4.1 root cause #5
> Status: PREVIEW — follow this procedure AFTER branch reconciliation is complete
>         (per plan §18 next-steps priority order).

## Prerequisites

Before creating the v3.0.0 tag, ensure:

1. ✅ Branch reconciliation complete — `scripts/branch-reconciliation.sh --apply` has run successfully and local `main` is in sync with `origin/main`
2. ✅ V3.0 features are deployed to production and smoke test passes
3. ✅ `scripts/backup-to-github.sh` exists at the repo root (promote from `preview/v3.0-plan-execution/scripts/`)
4. ✅ `gh` CLI is authenticated: `gh auth status` shows you logged in
5. ✅ The V3.0 tarball exists: `download/aisalon-massapro-V3.0.tar.gz` (or V3.1 — use the latest)
6. ✅ Release notes are reviewed: `preview/v3.0-plan-execution/github-release/v3.0.0-release-notes.md`

## Procedure

### Step 1 — Promote the backup script and release notes to the repo root

```bash
cp preview/v3.0-plan-execution/scripts/backup-to-github.sh scripts/
chmod +x scripts/backup-to-github.sh
cp preview/v3.0-plan-execution/github-release/v3.0.0-release-notes.md ./
```

### Step 2 — Verify clean working tree

```bash
git status --porcelain
# Should be empty (or only show the files you just copied, which you should commit)
```

If you just promoted the files above, commit them first:
```bash
git add scripts/backup-to-github.sh v3.0.0-release-notes.md
git commit -m "chore: promote backup script and v3.0.0 release notes"
git push origin main
```

### Step 3 — Run the backup script

```bash
./scripts/backup-to-github.sh v3.0.0 \
  --tarball download/aisalon-massapro-V3.1.tar.gz \
  --notes v3.0.0-release-notes.md
```

Notes:
- We use the V3.1 tarball (latest milestone backup) even though the tag is v3.0.0, because V3.1 includes the agenda thumbnail fix and is the most accurate snapshot of what's running in production.
- If you'd rather tag V3.1 separately, use `v3.1.0` as the tag name and adjust the release notes accordingly.

### Step 4 — Verify the release

```bash
gh release view v3.0.0
```

Should show:
- Tag: v3.0.0
- Title: v3.0.0
- Notes: contents of `v3.0.0-release-notes.md`
- Assets: `aisalon-massapro-V3.1.tar.gz`

### Step 5 — Verify on GitHub

Open https://github.com/EzeCaz/aff-massapro/releases/tag/v3.0.0 in a browser. Confirm:
- Release is published (not draft)
- Release notes render correctly
- Tarball asset is downloadable
- MD5 of downloaded asset matches `download/aisalon-massapro-V3.1.tar.gz`

### Step 6 — Update worklog

The backup script automatically appends a worklog entry. Verify it landed:

```bash
tail -20 worklog.md
```

### Step 7 — Clean up

Move the release notes out of the repo root (they're now in the GitHub Release):

```bash
mkdir -p docs/release-notes
mv v3.0.0-release-notes.md docs/release-notes/
git add docs/release-notes/
git commit -m "chore: file v3.0.0 release notes in docs/"
git push origin main
```

## Rollback (if something went wrong)

To delete the tag and release (within 5 minutes of creation, before anyone depends on it):

```bash
gh release delete v3.0.0 --yes --cleanup-tag
git tag -d v3.0.0  # in case it's still local
git push origin :refs/tags/v3.0.0  # in case it's still on origin
```

Use semantic versioning:
- **MAJOR** (X.0.0) — breaking changes (e.g., new database schema that's not backward-compatible)
- **MINOR** (0.X.0) — new features, backward-compatible
- **PATCH** (0.0.X) — bug fixes only
